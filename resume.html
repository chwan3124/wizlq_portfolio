    이를 위해 Next.js를 선택한 이유에 대해 체계적으로 문서화하여 개발 팀원들을 초청해 세미나를 진행하였습니다. 세미나에서는 기존 Nexacro로 개발된 백오피스의 단점을 설명 후 기존 백오피스의 일부 페이지를 Next.js로 구현한 예시와 함께 디렉토리 기반 라우팅, SSR 등 주요 개념과 장점을 다뤘습니다. 이를 통해 개발 팀원들의 Next.js 마이그레이션에 대한 긍정적인 반응과 Next.js에 대한 기본적인 이해를 갖출 수 있게 되었습니다.

문제 해결

  • 클라이언트 인증 상태 유지 문제
  • 프로젝트 초기엔 세션 기반 인증 방식을 도입하여 로그인 후 클라이언트의 인증 상태를 백엔드 서버 세션으로 관리했으나 서버 재시작 시 세션 데이터 손실되어 사용자가 재로그인이 해야 하는 문제가 발생하였습니다.

    이를 해결하기 위해서 Session Clustering이나 별도 세션 저장소 등을 활용해 세션 정보를 유지하는 방법을 검토했습니다. 하지만 당시 백엔드 서버는 단일 인스턴스로 구성되어 있었고 서버를 하나 이상 추가하여 모든 서버가 동일한 세션 정보를 공유하도록 설정할 경우 메모리 사용량이 많아질 것이라고 판단했고
    Redis 같은 저장소를 도입하여 세션 정보를 유지할 수는 있지만 추가적인 네트워크 비용과 Redis의 장애를 대비하기 위해 클러스터링 환경을 구성해야 하는 등 운영 복잡성이 증가한다는 문제가 있어 다른 방법을 알아보던 중 JWT 기반 인증 방식을 알게 되었습니다.

    JWT는 로그인 후 서버가 클라이언트에게 유효한 Access Token을 발급하고, 이후 클라이언트는 요청 시 전달되는 Access Token을 서버에서 유효성 검증만을 하는 구조로 동작되기 때문에 JWT 기반 인증 방식을 도입한 결과 백엔드 서버는 Stateless 구조로 전환되어 RESTful 아키텍처 원칙을 준수하는 등 확장성을 높일 수 있었고 서버가 재시작되어도 클라이언트는 인증 상태를 유지할 수 있게 되었습니다.
  • 로그 파일의 디스크 용량 문제
  • 로그 라이브러리로 사용된 Log4j2를 이용해 로그를 기록했지만 로그 파일이 지속적으로 증가함으로써 추후 디스크 용량을 초과할 것이란 문제가 있었습니다.

    이를 해결하기 위해 최소한의 로그가 기록될 수 있도록 INFO 레벨에서 DEBUG 레벨로 제한하고 로테이션 정책을 설정하여 매일 자정마다 새로운 로그 파일로 교체되도록 설정하고, 최대 7일 분량의 로그 파일만 유지되도록 구성했습니다.
    그 결과 7일 이전 로그 파일은 자동으로 삭제하면서 로그 파일의 크기를 유지하고 안정성을 확보할 수 있었습니다.
  • Table Full Scan 문제
  • 모든 쿼리문은 DB Optimizer의 실행 계획을 통해 인덱스 참조 여부를 확인하여 최적화된 쿼리를 작성했습니다. 하지만 퀴즈 검색 쿼리의 경우 제목과 내용을 검색 조건으로 LIKE %% 형태의 조건문 사용한 것이 Full Table Scan을 발생하는 것으로 확인되었습니다. 현재 데이터양은 많지 않았지만 데이터가 증가할 경우를 대비하기 위해 개선이 필요했습니다.

    이를 위해 Full Text Index를 고려했지만 Full Text Index를 참조하기 위한 쿼리 작성 방식이 추후 검색 조건의 변경 가능성으로 인해 복잡성이 존재했고 데이터가 많아질수록 그만큼 많은 계산 작업이 필요하기 때문에 단기적으로 성능 개선 효과를 볼 수 있지만 장기적으론 확장성과 유연성이 부족하다고 생각하였습니다.
    반면에 Elasticsearch는 DB와 독립적으로 동작하는 분산형 검색 엔진으로써 높은 확장성과 성능을 가지는 특징을 가지고 있기 때문에 검색 작업을 맡는 독립적인 엔진인 Elasticsearch를 도입하여 DB 부하를 줄이고, 다양한 검색 기능과 확장성을 높이는 것이 시스템 개선에 적합하다고 판단하여 Elasticsearch를 도입하게 되었습니다.

    약 100만 건의 공공 데이터를 대상으로 LIKE %% 검색과 Elasticsearch를 이용한 조회 성능 비교 결과 1800ms -> 45ms로 단축되어 약 97%의 성능 향상과 데이터양이 증가해도 검색 속도를 유지할 수 있는 결과를 얻었습니다.
  • 반복되는 사용자 정보 조회 로직 문제
  • API 개발 중 호출한 사용자의 userId를 조회하여 데이터만 변경 및 삭제하는 API들이 있었습니다. 비즈니스 로직 내에서 매번 인증된 사용자 정보를 조회하는 코드를 반복적으로 작성하니 유지보수성이 떨어지고 비효율적이라고 생각되었습니다. 공통 로직으로 분리할 수 있는 방법을 구글링으로 찾던 중 Argument Resolver를 이용하면 메소드 특정 타입의 파라미터에 데이터를 주입할 수 있는 방법이 있다고 하여 적용하게 되었습니다.

    ex)
    @PostMapping("/api/quiz")
    public ApiResponse regist(@RequestBody @Valid QuizRegReq req, /* 사용자 정보 조회용 어노테이션 */ UserInfo userInfo){ ... }

    먼저 특정 타입의 파라미터를 처리하기 위해 HandlerMethodArgumentResolver를 구현했습니다. 해당 Resolver는 SecurityContextHolder에 저장된 인증 정보를 가져와 사용자 정보 객체를 생성 후 return 하여 실제 파라미터에 주입하는 구조로 재구성한 결과 중복된 코드를 제거할 수 있었고 보다 쉽게 사용자 정보를 얻을 수 있어 유지보수성을 높일 수 있었습니다.
